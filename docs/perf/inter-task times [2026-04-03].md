### What causes outlier transitions

In previous analysis (6-mo baseline), Eyelink's edf2asc conversion was the dominant bottleneck at 16.5s/transition. That has since been deferred to end-of-session, bringing ET stop down to 0.71s.

Outlier transitions (>8s end-to-end) fall into two distinct patterns:

#### Pattern A: High stop time (FLIR queue drain)

The FLIR drain is **blocking** — `stop_recording()` must fully complete before `start_recording()` begins (`server_acq.py:147-160`). The chain: `ensure_stopped()` → `video_thread.join()` (no timeout) → record thread waits on `save_thread.join()` → save thread drains entire queue to disk.

The FLIR at 195fps queues frames faster than disk can write during some sessions. Affected sessions:

| Session | Task | Queue at stop | ACQ stop time |
|---------|------|-------------:|-------------:|
| 100867 | MOT_obs_1 | ~6,394 | 31.07s |
| 100867 | DSC_obs | — | 35.81s |
| 100311 | DSC_obs | — | 31.68s |
| 100801 | MOT_obs_1 | — | 20.34s |
| 100801 | saccades_h | ~3,858 (carryover) | 15.78s |

Note: only sessions 100801 and 100867 showed FLIR queue issues. Sessions 100226, 100766, and 101102 had normal FLIR drain times (<1s).

#### Pattern B: Normal stop, high start time (majority of outliers)

Most outliers have stop times of ~2-3s (normal) but device start times of 7-14s. These are NOT explained by FLIR queue drain. Examples:

| Session | Task | Stop | Start | Total |
|---------|------|-----:|------:|------:|
| 100226 | hevelius_obs | 3.00 | 10.31 | 13.31 |
| 100226 | foot_tapping | 2.54 | 10.60 | 13.14 |
| 100311 | picture_description | 2.66 | 10.88 | 13.54 |
| 100766 | coord_pause | 0.00 | 14.88 | 14.88 |
| 101102 | hevelius_obs | 2.55 | 10.28 | 12.83 |

Tracing the 100226 hevelius→break_video_2 transition in detail (stop=3.00, start=10.31):

```
13:39:23.195  ACQ: device stop completes (RealSense pipelines)
13:39:23.198  ACQ: frame_preview sent
13:39:23.872  ACQ: frame preview callback
              *** ~7 SECOND GAP ***
              (inside start_recording → device.start() for each camera)
13:39:30.816  CTR: first NewVideoFile arrives
13:39:31.025  ACQ: FLIR/RealSense begin recording
13:39:33.935  ACQ: last RealSense enters LSL loop
13:39:34.184  ACQ: device start took 10.31
```

The ~7s gap is inside `start_recording_devices()` (lsl_streamer.py:259-275), which uses a `ThreadPoolExecutor` to call `device.start(filename)` on each camera. The gap occurs before any "Beginning Recording" log appears, meaning it's in the pre-recording phase of device.start() — likely output file creation (video container setup, directory creation) and device initialization.

**This gap varies significantly**: 2-4s on some transitions, 7-10s on others, even for the same device set in the same session. The cause of this variation is not fully explained by the logs. Possible factors include disk I/O contention (from previous video file writes), memory allocation for video buffers, or RealSense USB initialization timing — but these are hypotheses, not confirmed.

### Open questions

1. **Why does the device start pre-recording phase vary 2-10s** for the same device configuration? The logs don't capture what happens inside `device.start()` before "Beginning Recording" is logged. Adding timing instrumentation inside `start_recording_devices` (per-device) would help isolate whether it's one device or all devices that slow down.

2. **Does disk I/O from previous FLIR writes cause the slow starts?** The FLIR writes large video files (195fps × task duration). Even without queue backlog, the OS may still be flushing buffers to disk during the next transition. This could explain why longer tasks are followed by slower starts.

3. **Why do only some sessions develop FLIR queue backlog?** Sessions 100226 and 101102 had clean FLIR drains throughout, while 100801 and 100867 had severe backlog. The resource data doesn't show dramatic differences. Disk write rate or background process activity could differ but isn't captured in the current logs.
